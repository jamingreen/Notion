Entity

An entity is an item of interest about which information is stored

Flat file

A flat file is a database that consists of a single file. The flat file will most likely be based around a single entity and its attributes.

  • What problem might it arise?
    • Data might be inconsistent
      • For example the amount of cartridge in one printer might be changed in one record but not changed in another

Relational database

A relational database is a database which recognises the differences between entities by creating different tables for each entity.

  • Different in relational database and flat file
    • In flat file, all record are stored in one table
      • In relational database, multiple table is used
    • Data redundancy
      • Space wasted

Relational Database

Primary Key

It is a unique identifier for each record in the table.

Should be shown by underlining

Foreign Key

It is the attribute that links two tables.

A primary key from one table used as an foreign key in another table.

Should be should by asterisk (*)

Secondary Key

An non-unique attribute which is indexed for fast lookup It is used in search and ordering

Composite Key

A combination of keys that is unique

A key that consist of 2 attributes

Entity Relationship Modeling

One-to-one

Each entity will only link to one other entity in another table

One-to-many

One entity can be associated with multiple entities

Many-to-many

One entity in table A can related to multiple entities in table B, while one entity in table B can also relate to multiple entities in table B

Relation graphs

Normalisation

The process of coming up with the best possible layout for a relational database

  • No redundancy
  • Consistent data
  • Records can be added and removed without issues
  • Complex query can be carried out

Zero Normal Form (0NF)

Have repeating group that have merge rows

Multiple observation / variable in a column

First Normal Form (1NF)

Have duplication of data (divide the merge rows from 0NF)

  • No repeat attribute
  • All attribute are atomic (Cannot be further broken down)
  • All attribute has a unique name
  • Each row has a unique primary key

Second Normal Form (2NF)

  • First Normal Form
  • no partial dependency
    • Partial dependency — A attribute depends on part of a composite key

Third Normal Form (3NF)

  • Second Normal Form
  • no non-key dependency
    • Non-key dependency — A attribute depends on a attribute that is not a key
  • no transitive key dependency
    • Transitive dependency — A attribute depends on a key (through composite key) that has no relation in reality

Dependencies

  • Partial Dependency

    pAddress only depends on propertyNo but no iDate

  • Non-key Dependency

    sName (Staff name) depends on staffNo (which is not a key)

  • Transitive Dependency

    sName (Staff name) depends on iDate (iDate doesn’t directly imply the staff’s name)

Referential Integrity (changed)

A table is linked to another table using primary key and foreign key. The foreign key value must have a corresponding record with that as primary key in another table.

When the record with a primary key is deleted, the records in another table with that as foreign key must also be deleted.

Ensure consistency

This ensure that information is not removed if it is required elsewhere in a linked database

A table cannot be deleted if it has primary key that is linked to another table

Transaction Processing

It is a single or a series of operations executed on data

ACID (Atomicity, Consistency, Isolation, Durability)

Criteria of the database

  • Atomicity
    • A transaction is processed in its entirety or not at all
    • No incomplete transaction
  • Consistency
    • No transaction can violate any of the defined validation rules
    • Referential integrity (specified in the set up of the database) must be upheld
      • If the primary key record is deleted, any record with that foreign key must be deleted
    • Table must joined by the primary and foreign key as indicated in the set up
  • Isolation
    • It ensures concurrent execution of transaction leads to the same result as if there were processed one after the other
  • Durability
    • Once a transaction has been committed, it will remain so, even in the event of a power cut
    • After each part of the transaction is complete, it is held in a buffer until all elements of the transaction is executed
    • Only then will the real database be change

Record Locking

It forbidden simultaneous access to records.

It is used to prevent inconsistency or loss of updates

Deadlock

When 2 user request access to the other table (but since tables are accessed, both cannot finish their transaction until the table is unlocked)

Serialisation

Ensures transactions do not overlap in time and thus cannot interfere with each other

Timestamp ordering

Read timestamp, write timestamp (There are global for all users)

If the read timestamp is not the same as the one when they start the transaction, the DBMS knows that another transaction is done, notify the user and block the transaction

Commitment ordering

The transactions are ordered in terms of their dependency on one another

  • No transaction will be lost

Redundancy

Duplicate hardware with same information located in different geographical / physical location.

It act as a backup

It mirrors every transaction on the main server

It can automatically take over in the event of hardware failure
